# Mnemonic Skill Residency

Status: hardened implementation guide.
Date: 2026-05-14

## Purpose

Skill Residency turns `.arlecchino/skills/**/SKILL.md` files into compact,
trusted, project-local procedural context. The agent should use a small resident
skill packet once a skill has been reviewed and activated, instead of repeatedly
loading full skill bodies.

Resident skills are advisory context only. They never grant tool permissions,
never override the latest user instruction, and never outrank safety policy,
provider consent, MCP approvals, or repo `AGENTS.md`.

## Hard Invariants

- One process-local SQLite owner opens `.arlecchino/ai/mnemonic.db`; Mnemonic
  facts and skill residency are separate typed domains on that shared database.
- Skill residency never uses `mnemonic_entries` as its source of truth.
- Read paths are side-effect-free: `list`, `search`, `context`, and `status`
  must not rewrite files, rebuild digests, update access counters, or sync
  generated context files.
- Imported, generated, MCP-derived, or newly discovered project skills start as
  candidate/untrusted and cannot enter provider context until an explicit
  review/pin path promotes them.
- Full `SKILL.md`, `references/`, `scripts/`, and command recipes must not be
  sent to providers. Provider context receives compact trusted digests only.
- Skills expose tool hints only. They do not carry allowlists, grants, approval
  bypasses, or executable permission policy.
- Runtime residency is keyed by workspace root hash, agent surface, session
  instance, skill hash, digest version, state, and TTL. It is not keyed by the
  stable AI project session id alone.
- Disabling Mnemonic prevents AI/MCP reads and writes from using the shared
  context path; clear/state reset must also clear resident runtime state.

## Current Backend Shape

- `internal/ai/storage` owns the shared SQLite handle, WAL settings, migration
  versioning, Mnemonic schema, and skill residency schema.
- `internal/ai/mnemonic` stores durable facts. Search/list paths are pure reads,
  and MCP-saved facts are generated by default rather than trusted.
- `internal/ai/skills` stores the typed registry, reviewed digests, residency
  state, events, import quarantine metadata, and hash invalidation.
- `internal/ai` can include trusted active skill digests in
  `AIContextSnapshot.Skills` when `IncludeSkills` is requested and Mnemonic is
  enabled. Privacy sanitization and byte budgeting cover skill fields.
- `internal/mcp` exposes `agent_skills.list`, `agent_skills.status`, and
  `agent_skills.context` as read-only tools. `agent_skills.pin`, `activate`,
  `dismiss`, and `import` are approval-gated.
- Terminal guide generation points agents to `agent_skills.context` and forbids
  raw project skill files in provider context. `.arlecchino/memory/CONTEXT.md`
  remains memory recall, not skill authority.

## Review And Activation Flow

1. Project open or MCP service startup scans `.arlecchino/skills/**/SKILL.md`
   and records candidate skill metadata plus `contentHash`.
2. `agent_skills.list/status` can inspect candidates without reading full skill
   bodies into the prompt.
3. A user-approved review path pins a skill and creates a deterministic compact
   digest. Suspicious approval, credential, or off-root execution instructions
   are filtered from operating reminders.
4. Activation writes short-lived residency state with TTL. Context reads join
   active/resident state to the current trusted digest and current content hash.
5. If the skill file hash, digest version, trust state, enabled state, or clear
   state changes, stale resident context drops out instead of surviving silently.

## Public Interface Notes

- `AIContextRequest.includeSkills` requests resident skill context.
- `AIContextSnapshot.skills` contains compact digest fields only.
- `AIContextSummary.skillCount` reports included skill digests.
- MCP tools:
  - `agent_skills.list` and `agent_skills.status`: inspect registry/status.
  - `agent_skills.context`: compact trusted resident context only.
  - `agent_skills.pin`, `agent_skills.activate`, `agent_skills.dismiss`,
    `agent_skills.import`: approval-gated writes.
- `ide_control.request_permission` must reject unknown or disabled tool names.

## Verification Checklist

- DB: shared handle/refcount, migration records, skill tables, scoped clear, no
  FTS rebuild on every open.
- Mnemonic: search/list do not update access metadata; disabled store blocks
  reads and writes while clear still works.
- Skills: candidates do not produce context; reviewed/pinned skills do; hash
  changes invalidate active residency.
- AI: untrusted skills stay out of snapshots; trusted resident digests are
  sanitized, budgeted, and rendered without full `SKILL.md` body text.
- MCP/terminal: `initialize` stays static, skill read tools are pure, write
  tools are approval-gated, and terminal bootstrap prefers `agent_skills.context`.

## Deferred Work

- Read-only frontend inspector for registry/status once the backend contract is
  stable and Wails bindings are regenerated through the normal flow.
- Future AI chat panel must use generated backend bindings for
  `AIContextRequest.includeSkills`, `AIContextSnapshot.skills`, and
  `AIContextSummary.skillCount`. It must not read raw project skill files,
  references, scripts, or command recipes into frontend state or provider
  context.
- Chat UI can expose context disclosure and included skill counts, but skill
  residency remains advisory context only. Tool approval, activation/review, and
  import trust promotion stay in dedicated backend/user-review flows rather than
  hidden chat shortcuts.
- User/global/plugin skill roots after project-local behavior is proven.
- Model-assisted digest generation only after deterministic digests and user
  review are reliable.
- Marketplace/install UX with quarantine, provenance, preview, and explicit
  trust promotion.
